# TextRankによるキーワード抽出の詳細フロー

本プロジェクトでは、APIの英語説明文から特徴量となるキーワードを抽出するために、**TextRankアルゴリズム**（spaCy + pytextrank）を利用しています。

ここでは、その具体的な処理のながれを実例とともに詳説します。

---

## 全体像: TextRankアルゴリズムとは

TextRankは、Googleの検索エンジンで使われている「PageRank」アルゴリズムをテキストに応用したものです。
Webページ間のリンク関係の代わりに、**「単語間の共起関係（近くに同時に出現するかどうか）」をネットワーク（グラフ）として構築**し、そのネットワークの中で多くの繋がりを持つ「中心的な単語」を重要キーワードとして抽出します。

---

## 処理のながれ（3ステップ）

実際のコード（`src/make_keyword.py`）では、以下の3ステップで処理が行われます。

### 例題: `CreateFileW` の場合
API説明文: *"Create or open a file or device using Unicode name"*

### Step 1: 形態素解析とフィルタリング
まず、spaCyを用いて文章を単語（トークン）に分割し、品詞（名詞・動詞など）を判定します。

| 元の単語 | 品詞 | 判定結果 | レンマ（基本形） |
|---|---|---|---|
| Create | 動詞 (VERB) | 🟢 採用 | `create` |
| or | 接続詞 (CCONJ) | ❌ 除外（ストップワード） | - |
| open | 動詞 (VERB) | 🟢 採用 | `open` |
| a | 冠詞 (DET) | ❌ 除外（ストップワード） | - |
| file | 名詞 (NOUN) | 🟢 採用 | `file` |
| or | 接続詞 (CCONJ) | ❌ 除外（ストップワード） | - |
| device | 名詞 (NOUN) | 🟢 採用 | `device` |
| using | 動詞 (VERB) | 🟢 採用 | `use` |
| Unicode | 固有名詞 (PROPN) | 🟢 採用 | `unicode` |
| name | 名詞 (NOUN) | 🟢 採用 | `name` |

この段階で、意味を持たないストップワード（or, a）や句読点は除外され、**名詞・固有名詞・動詞**だけが抽出されます。また、複数形や過去形はすべて基本形（レンマ）に統一されます。

### Step 2: 共起グラフの構築
抽出された単語のリスト `[create, open, file, device, use, unicode, name]` を使ってネットワークを作ります。

TextRankでは「指定されたウィンドウサイズ（例: 3単語以内）」に同時に出現した単語同士を線（エッジ）で結びます。

- `create` と `open` は近いから結ぶ ──
- `open` と `file` は近いから結ぶ ──
- `file` と `device` は近いから結ぶ ──

このようにして、単語を「ノード」、近接出現を「エッジ」とした巨大なネットワークグラフが出来上がります。

### Step 3: 中心性計算（PageRank）
ネットワーク全体の中で、**「どの単語が一番多くの（かつ重要な）単語からリンクされているか」**を計算します。

例えば、`file` という単語は：
- `CreateFileW` の説明文内でも `open` や `device` と繋がる
- 他のAPI（`ReadFile`, `DeleteFileW`）の説明文でも、`read`, `write`, `delete` など様々な動詞と結びつく

結果として、**`file` にはネットワーク全体から無数のエッジが集中する**ことになります。
これが「PageRankにおける中心性が高い」という状態です。

pytextrankは、この中心性スコアが高い順にフレーズをランク付けします。本システムではその**上位3つのフレーズ**を構成する単語をカウントし、最終的なキーワードとして出力します。

結果: `CreateFileW` → `["file", "device", "name"]`

---

## 課題: なぜ「機能フィルタ（方針B）」が必要だったのか？

このTextRankの仕組み（共起に基づく中心性計算）には、1つ弱点があります。
それは**「統計的なつながりが強いだけで、マルウェア解析的な意味がない単語」も高く評価してしまう**ことです。

### 具体例: `life_cycle` 問題
MicrosoftのAPIドキュメントには、以下のような決まり文句（ボイラープレート）が頻繁に登場します。
- *"Manages the life cycle of the COM component"*
- *"Initializes the state of the object life cycle"*

するとどうなるでしょうか。
`life`, `cycle`, `state` といった単語は、**様々なAPIのドキュメントで毎回セットで登場し、常にお互いを強く結びつけ合う（共起する）**ため、ネットワークの中心性が異常に高くなってしまいます。

その結果、TextRankアルゴリズムは `life_cycle` を「超重要キーワード」だと判定してしまいます。しかし、マルウェアの動作解析において「COMコンポーネントのライフサイクル」という単語は、ファイル操作や通信といった具体的な挙動とは結びつかない「ノイズ」です。

### 解決策
この「NLPアルゴリズムとCTI（脅威インテリジェンス）のギャップ」を埋めるために用意したのが、**機能直結キーワードフィルタ（方針B）**です。
統計的に抽出されたキーワード群に対して、CTI視点で手動で作成した「11の機能カテゴリ（ホワイトリスト）」を通すことで、`life_cycle` や `option` といった単語を機械学習やSHAPの解釈から排除しています。
